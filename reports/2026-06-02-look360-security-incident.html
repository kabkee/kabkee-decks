어떤 공격이었나 SQL 인젝션

🍽️ 비유로 이해하기
식당에서 손님이 주방에 주문서를 보냅니다. 정상 손님은 "콜라 한 잔"이라고 적죠. 그런데 공격자는 주문서에 "콜라 한 잔 — 그리고 금고를 열어라" 처럼 명령을 몰래 섞어 적습니다. 시스템이 이걸 진짜 명령으로 착각하고 실행하면, 저장된 정보가 새어나갈 수 있습니다.
SQL 인젝션 : 입력칸에 데이터 대신 데이터베이스 명령어를 몰래 넣어, 시스템이 그 명령을 실행하게 만드는 대표적인 웹 공격.
sqlmap : 이 공격을 자동으로 수천 번 반복 시도하는 해킹 도구. 이번 공격에 사용됨 (총 약 1.2만 건의 공격 요청).

왜 사이트가 멈췄나 과부하의 원리

📞 비유로 이해하기
서버에는 손님을 응대하는 상담원(처리 일꾼)이 정해진 수만큼 있습니다. 공격자는 "5초 동안 가만히 기다려" 라는 명령을 수없이 보냈고, 그때마다 상담원 한 명이 아무것도 못 하고 묶여버렸습니다. 결국 모든 상담원이 붙잡혀 정상 손님(이용자)을 받을 수 없게 된 것입니다.
  • 이 서버는 처리 일꾼이 최대 200개지만, 실제 CPU는 2개뿐이라 과부하에 취약했음
  • "기다려" 공격이 일꾼 + 데이터베이스를 동시에 묶어 완전 정지

우리 데이터는 안전한가? 가장 중요한 질문

✅ 안전 — 유출 안 됨

  • 현재 활성 회원 정보 전체
  • 관리자 계정 정보
  • → 공격이 거기 도달하기 전에 차단되어 막힘

⚠️ 노출 정황 — 유출 가정

  • 휴면회원 이메일 약 292건
  • 휴면회원 비밀번호 약 193건
    (암호화된 형태, 평문 아님)
  • 이름·전화는 휴면처리로 대부분 이미 삭제됨
휴면회원 : 오랫동안 로그인하지 않아 별도 보관 중인 계정. 개인정보보호를 위해 이름·연락처 등은 이미 상당수 파기된 상태였음. 노출된 비밀번호도 암호화(해시)되어 있어 즉시 악용은 어렵지만, 안전을 위해 유출됐다고 가정하고 대응합니다.

무엇이 문제였나 근본 원인

  • 🚪 입력값 검사가 약했다 — 검사 장치는 있었지만, 일부 위험 단어만 걸러내는 허술한 방식이라 공격자가 우회함 "위험 단어 목록"만 막는 방식은, 목록에 없는 표현·대문자·따옴표로 쉽게 빠져나갈 수 있음
  • 🔑 데이터베이스 접속 권한이 과했다 — 앱이 최고 권한(root)으로 접속해, 뚫렸을 때 피해 범위가 커질 수 있었음
  • 🌐 서버가 외부에 직접 노출 + 자동 차단 장치(방화벽 규칙) 부재로 공격을 초기에 막지 못함
핵심: "공격을 당해서"가 아니라, 입력을 안전하게 처리하지 못하는 구조가 근본 원인입니다. 공격은 그 약점을 건드린 방아쇠일 뿐입니다.

이미 완료한 긴급 조치 당일 ~ 후속

  • 🛡️ 공격 패턴·공격자 IP 즉시 차단 — 같은 공격 재유입 방지, 서비스 복구
  • 🔒 데이터베이스 외부 접속 완전 차단 — 외부에서 DB에 직접 접근 불가하도록 방화벽 영구 설정
  • 🔑 앱 접속 권한 최소화 — 최고 권한(root) 대신 꼭 필요한 권한만 가진 전용 계정으로 교체
  • 🗝️ 위험 계정 제거 + 비밀번호 교체 — 외부 노출돼 있던 관리 계정 삭제, 비밀번호 새로 발급 후 안전 보관
  • 🔍 유출 범위 정밀 조사 완료 — 무엇이 노출됐는지 로그로 확인 (위 6페이지)

앞으로 할 일 재발 방지

  • 🧱 근본 수정 — 입력을 명령과 완전히 분리하는 안전한 코드 방식(프리페어드 스테이트먼트)으로 전환 위험 단어를 "거르는" 방식이 아니라, 입력이 애초에 명령이 될 수 없게 만드는 근본 해법
  • 📧 휴면회원 보호 조치 — 노출 정황 계정 비밀번호 무효화 + 개인정보보호법상 통지 의무 검토
  • 🌐 방어막 강화 — 외부 보안 필터(WAF) 적용, 서버 직접 노출 차단, 과부하 대비 설정 조정
