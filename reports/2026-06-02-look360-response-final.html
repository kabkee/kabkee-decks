① 무슨 일이 있었나 사고 · 1차대응

🌙 2026-06-02 새벽, 외부(중국 IP)에서 자동화 해킹 도구(sqlmap)로 사이트 주소에 SQL 인젝션 공격을 대량 유입. "5초 기다려"류 공격이 서버 처리 일꾼을 모두 묶어 05:58 전면 다운.
  • ⚡ 1차 대응 — 공격 패턴·공격자 IP 즉시 차단 + 서버 재시작 → 08:20경 정상 복구
  • 📌 여기까지가 응급 조치. 이후 "왜 뚫렸나·무엇이 새어나갔나"를 정밀 분석
SQL 인젝션 = 입력칸에 데이터 대신 명령어를 몰래 섞어 시스템이 실행하게 만드는 공격. sqlmap = 이를 자동 반복하는 해킹 도구.

② 진짜 원인을 찾다 정밀 분석

"공격당해서"가 아니라 — 입력값을 안전하게 거르지 못한 코드 구조가 근본 원인.
  • 🔍 입력을 막는 검사 장치는 있었지만, 위험 단어 몇 개만 지우는 허술한 방식이라 대문자·따옴표·시간지연 명령을 그대로 통과시켰음 "위험 단어 목록"만 막는 방식은 빠져나갈 구멍이 끝없이 생김
  • 🔑 앱이 DB에 최고 권한(root)으로 접속해, 뚫렸을 때 피해 범위가 클 수 있었음
  • 🌐 서버가 외부에 직접 노출 + 자동 차단장치 부재로 초기에 못 막음

③ 데이터는 안전한가? 유출 포렌식

✅ 안전 — 유출 안 됨

  • 현재 활성 회원 전체
  • 관리자 계정
  • → 공격이 도달하기 전에 차단되어 막힘

⚠️ 노출 정황 (제한적)

  • 휴면회원 이메일 — 실질 182건
    (전체 292건 중 110건은 회사 생성 더미)
  • 비밀번호는 암호화 형태(평문 아님)
  • 이름·전화는 이미 대부분 파기됨
접속 기록 전수 분석으로 확인 — 차단 타이밍 덕에 현재 회원·관리자는 무사. 노출 정황은 휴면 일부로 한정.

④ 보안 강화 조치 완료

  • 🔒 DB 외부 접속 완전 차단 — 방화벽 영구 설정 (외부에서 DB 직접 접근 불가)
  • 🔑 앱 접속 권한 최소화 — 최고권한(root) → 꼭 필요한 권한만 가진 전용 계정으로 교체
  • 🗝️ 위험 계정 제거 + 비밀번호 교체 — 외부 노출돼 있던 과잉권한 계정 삭제, 새 비밀번호는 안전 금고(Vaultwarden) 보관
  • 🔍 유출 범위 정밀 조사 완료 — 무엇이·얼마나 노출됐는지 로그로 규명

⑤ 시스템 개선 — 휴면계정 정책 개편·배포

개인정보보호법 개정(휴면제 폐지)을 반영해, 오래 방치된 개인정보를 최소화하는 방향으로 정책을 개선. 보관할 개인정보가 적을수록 = 유출돼도 피해가 작다는 보안 원칙과도 일치.
  • 🗑️ 1년 이상 미접속 회원 → 휴면 전환 + 개인정보 즉시 파기 (재접속 식별용 닉네임만 유지)
  • 🛡️ 휴면 처리 기능은 관리자만 실행 가능하도록 잠금 (외부 무단 실행 차단)
  • 🧹 더 이상 쓰지 않는 개인정보 분리보관 테이블 비움 (불필요 데이터 제거)
  • ✅ 코드 검증 후 운영 서버 배포 완료

⑥ 근본 방어 — SQL 인젝션 3중 차단 강화·배포

이번 공격의 통로였던 SQL 인젝션을, 한 겹이 뚫려도 다음 겹이 막도록 3중(다층)으로 방어 구축. 전수검사로 DB 접근 4,000여 곳을 점검해 적용.

🛡️ 3중 방어막

  • 1겹 — 입구(WAF): CloudFront에 SQLi 차단 룰 추가 (관찰모드)
  • 2겹 — 서버 관문: 모든 DB 접근을 한 곳에서 검사·차단 (419곳 일괄)
  • 3겹 — 코드: 위험 지점 29개 파일 개별 정제(입력값 안전처리)

✅ 안전장치

  • 실서버 복제 환경에서 실제 동작 검증(QC) — 정상기능 영향 0
  • 검증 중 발견한 잠재버그 3건도 수정
  • 관찰모드로 시작 → 오탐 없음 확인 후 차단 전환(무중단)
