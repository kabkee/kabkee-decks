🌐 전세계 독립형 360 VR 콘텐츠
수집 → external_vr DB 연동

krpano · Pano2VR 기반 공개 투어를 자동 발굴하여
external_vr 테이블에 적재 → Look360 플랫폼 연동

2026-05-26  ·  Kabkee

🤔 왜 필요한가?

  • 전세계 독립 서버에 배포된 krpano · Pano2VR 기반 360 VR 투어가 수십만 개 이상 파편화 존재
  • YouTube / Matterport 등 플랫폼 콘텐츠는 임베딩·재가공 불가 — 직접 배포된 공개 투어만 타겟
  • 현재 external_vr DB는 수동 입력에 의존 → 데이터 증가 속도가 느림
  • 자동 발굴 파이프라인을 갖추면 → Look360에서 더 많은 외부 VR 콘텐츠 노출 가능
🎯 "수동 입력을 줄이고, 검증된 외부 투어 데이터를 빠르게 쌓는 것"

🔭 타겟 탐지 원리 — 기술적 흔적 역추적

  • 무차별 크롤링 대신 VR 엔진이 남기는 고유한 코드 흔적만 타겟
  • 탐지 신호 예시:
    embedpano() 함수 호출  ·  krpano.js 스크립트 로드  ·  tour.xml 파일 참조
  • 이 신호가 있는 페이지 = 높은 확률로 360view 타입 투어 → 정밀 수집
  • 탐지 루트 3가지: Google Dorking · Censys/Shodan 서버 스캔 · Common Crawl 전수조사

⚠️ 한계 & 제약

넘어야 할 벽과 해결 방법

🔧 기술적 한계

  • Google/Bing 전체 인덱스 접근 불가  핵심 제약
    → Censys(현재 활성 서버) + Common Crawl(과거 전수) 조합으로 보완
  • Playwright 자원 소모 — 브라우저 실행이 무거움
    → 동시 5~10 탭, 타임아웃 15초 제한으로 관리
  • Dead Link 비율 — 독립 호스팅은 사이트 폐쇄 잦음
    → 90일 미응답 항목 자동 비활성화 배치 필요
  • 중복 URL — 동일 투어가 여러 도메인에 미러링
    → url 컬럼 UNIQUE 제약 + 정규화 처리

⚖️ 운영 · 법적 제약

  • robots.txt 준수  필수
    → 수집 전 robots.txt 파싱 — 금지 경로 자동 스킵
  • 이미지 저작권 — 원본 이미지/파노라마 파일 저장 불가
    → 스크린샷 썸네일만 저장, 원본은 원본 URL 링크로 처리
  • 검토 병목 — 수만 건이면 관리자 검토 공수 폭발
    → AI 신뢰도 점수 기준: 高신뢰 항목 자동 통과 / 低신뢰만 수동 검토
  • DB 연동 방식 결정 필요
    → 옵션 A: 기존 PHP 관리 화면 임포트 기능 추가
       옵션 B: look360_theme_external_vr API Lambda로 직접 INSERT
       옵션 C: DB 직접 접속 Python 스크립트 (가장 단순)
